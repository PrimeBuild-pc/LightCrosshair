# PresentMon Offline Validation Spike

Milestone 10 adds offline parsing and summary logic for PresentMon CSV-like
exports. It does not launch PresentMon, start ETW sessions, install binaries,
wire a runtime provider into LightCrosshair, or make user-facing verified
frame-generation claims.

## Manual Capture Outside LightCrosshair

Use a user-installed PresentMon build outside the app. Keep the capture
read-only and save CSV output for offline analysis.

Typical manual flow:

1. Install or unpack PresentMon from a trusted source.
2. Start the target game or sample renderer.
3. Capture a short run with the PresentMon console or capture application.
4. Enable frame-type tracking only when the PresentMon version and target stack
   support it.
5. Save the CSV output next to the test notes.
6. Feed the CSV into the offline parser from tests or a future research tool.

Do not add PresentMon binaries to this repository and do not make the
LightCrosshair app depend on PresentMon for this spike.

## Columns To Inspect

Useful columns vary by PresentMon version and capture mode. The offline parser
recognizes common names and ignores unknown columns.

| Area | Example columns |
| --- | --- |
| Target identity | `Application`, `ProcessID`, `ProcessName` |
| FPS | `FPS`, `Displayed FPS`, `Presented FPS`, `FPS-App`, `FPS-Presents`, `FPS-Display` |
| Frame/present timing | `FrameTime`, `MsBetweenPresents`, `MsBetweenDisplayChange`, `MsBetweenDisplayChangeActual`, `MsBetweenAppStart` |
| Presentation mode | `PresentMode` |
| Presentation flags | `AllowsTearing`, `PresentFlags`, `SyncInterval` |
| Dropped/late indicators | `Dropped`, `DroppedFrames`, `Late` |
| Frame generation | `FrameType` |

PresentMon documentation describes `FrameType` as the field that can identify
whether a displayed frame was rendered by the application or generated by a
driver/SDK when frame-type tracking and provider instrumentation are available.

## What LightCrosshair Can Infer Offline

The offline summary can compute, when columns exist:

- sample count;
- most common application/process name;
- average FPS;
- average frame or present interval;
- p95 and p99 frame time;
- present mode distribution;
- counts for allows-tearing, dropped, and late samples;
- explicit generated/interpolated frame sample count when `FrameType` is
  present and contains recognized generated/interpolated values.

All values are validation aids. They do not change runtime behavior.

## What LightCrosshair Must Not Infer

The offline parser must not classify frame generation as verified from:

- FPS ratios;
- alternating cadence;
- high displayed FPS;
- present timing anomalies;
- driver profile settings;
- game settings text;
- RTSS frame time;
- missing `FrameType` columns.

Those cases are `HeuristicOnly`, `Inconclusive`, or `Unavailable`, depending on
the available columns. `VerifiedSignalPresent` requires a dedicated recognized
column and value from the captured data, currently `FrameType` with a generated
or interpolated frame value.

## Why Runtime Integration Remains Blocked

Runtime integration still needs separate decisions:

- exact PresentMon version and license/notice review;
- user-installed dependency vs bundled binary vs library/service integration;
- ETW/admin behavior as standard user and admin;
- overhead on high-refresh displays;
- process targeting and lifecycle management;
- failure behavior if PresentMon is missing, blocked, or returns partial data;
- claim-guard tests for runtime wording;
- manual QA across native, capped, DLSS-G, FSR FG, AFMF, and unsupported cases.

Until those gates pass, LightCrosshair remains an out-of-process overlay with
ETW-style present telemetry, optional RTSS read fallback, heuristic
frame-generation suspicion, and a no-op frame limiter scaffold.

## Recommended Next Step

Milestone 11 should add a research-only command or test fixture workflow that
loads real captured PresentMon CSV files from a local ignored folder, compares
known off/on generated-frame scenarios, and records which columns are reliable
for each PresentMon version and GPU/driver stack. It should still avoid runtime
provider wiring.
